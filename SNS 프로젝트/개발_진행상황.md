### DB 스키마 (Supabase, `tmi_` 접두사, 전 테이블 RLS 적용)
- 테이블: `tmi_users`, `tmi_schedules`, `tmi_groups`, `tmi_group_members`, `tmi_appointments`, `tmi_appointment_participants`
- 트리거: `tmi_on_auth_user_created`(회원가입 시 프로필 자동 생성), `tmi_on_group_created`(그룹 생성자 자동 admin 등록)
- RPC(보안 정의자 함수): `tmi_join_group_by_code`, `tmi_get_group_calendar`, `tmi_find_common_time`, `tmi_create_appointment`, `tmi_is_group_member`
- 설계 원칙: `tmi_schedules`는 테이블 레벨에서 본인만 접근 가능 → 그룹 조회/공통시간탐색은 RPC로만 우회 접근, 비공개 일정 내용이 원천적으로 노출되지 않음
- 스토리지: `tmi-images` 버킷(공개, 프로필 이미지 업로드용)

## 2차 개발 — ❌ 미착수
- 소셜 로그인 확대 (Google / Kakao / Apple)
- 월간/주간 캘린더 전환
- 반복 일정
- 약속 수정 및 취소
- 참석/미정/불참 상태 관리 (DB 컬럼(`tmi_appointment_participants.status`)은 이미 존재, UI만 없음)
- 푸시 알림 / 알림 페이지 (DB 테이블 `tmi_notifications`는 아직 생성 안 됨)
- 일정 검색
- 그룹 대표 이미지
- 즐겨찾기 그룹 (DB 컬럼(`tmi_group_members.is_favorite`)은 존재, UI 토글 없음)
- 관리자/일반 멤버 권한 구분 (DB `role` 컬럼은 존재, 권한별 기능 분기는 없음)
- 공통 시간 없을 때 대체 시간 제공

## 3차 개발 — ❌ 미착수

## 다음에 이어서 개발할 때 참고사항
1. **배포 시크릿**: GitHub Actions 빌드에 `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` 레포 시크릿이 필요함. 이미 등록되어 있으나, 레포를 새로 만들 경우 반드시 push 이전에 먼저 시크릿부터 등록할 것 (순서가 바뀌면 빌드에 빈 값이 들어가는 문제가 실제로 발생했었음).
2. **base path**: `vite.config.js`의 `base`와 `main.jsx`의 `<BrowserRouter basename>`이 반드시 저장소명(`/T.M.I_sns/`)과 일치해야 함.
3. **이메일 인증 설정**: Supabase Auth의 "Confirm email" 옵션에 따라 실제 신규 가입자는 가입 직후 로그인이 안 될 수 있음 (시드 유저는 `email_confirmed_at`을 직접 채워 넣어 우회함).
4. **2차 기능 착수 시**: DB에 이미 준비된 컬럼(`status`, `is_favorite`, `role`)을 최대한 활용하면 스키마 변경 없이 UI만 추가하면 되는 항목이 많음.
5. **알림(2차) 착수 시**: `tmi_notifications` 테이블이 아직 없으므로 새 마이그레이션 필요 (기획서의 알림 유형: 그룹초대/약속생성/약속수정/약속취소/참여요청).
